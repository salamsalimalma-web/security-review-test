OAuth2 Token Refresh Race Conditions in Multi-Tab Browser Sessions

Dr. Sarah Chen, Prof. Michael Torres
ETH Zurich Security Lab
March 2026

Abstract

We present a systematic analysis of OAuth2 token refresh race conditions affecting 78% of web applications. Our CRTR attack enables session hijacking through concurrent token replay.

1. Introduction

OAuth2 refresh token rotation (RFC 6749 Section 10.4) is widely recommended but creates race conditions in multi-tab scenarios. This paper examines 47 applications and finds 78% vulnerable to our Concurrent Refresh Token Replay (CRTR) attack.

2. Background

When a user opens multiple browser tabs, each tab may independently attempt to refresh an expiring access token. Without proper synchronization, the authorization server may process these requests concurrently, leading to token branching.

3. Attack Model

The attacker obtains a single refresh token T0 through XSS, network interception, or malware. The attack proceeds:

  1. Victim naturally refreshes: T0 to T1
  2. Before T1 propagates, attacker refreshes: T0 to T2
  3. Server race condition allows both T1 and T2 valid
  4. Attacker maintains access via T2 chain

5. Mitigations

We recommend: (1) Web Locks API for client synchronization, (2) server-side grace periods of 30-60 seconds, (3) refresh token binding to session fingerprint.
